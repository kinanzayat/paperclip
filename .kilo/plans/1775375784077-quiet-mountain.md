# Paperclip: Linear-Like Experience Plan

**Date:** 2026-04-05
**Goal:** Transform Paperclip's board UI into a Linear-like team collaboration tool with chat-style comments, custom statuses, secure invite system with admin approval, user profiles, and a proper Admin/Member role model.

---

## Current State Summary

| Feature | Current Status | Gap |
|---------|---------------|-----|
| **Issue comments** | Timeline-based, supports markdown, @mentions, image attachments, agent + user authoring, real-time via WebSocket invalidation | UI needs polish for conversational feel; no reply threading |
| **Issue statuses** | 7 hardcoded statuses at every layer (DB, shared, server, UI) | No custom statuses; need full custom workflow system |
| **Labels** | Company-scoped, create/delete/assign via UI | No rename/recolor; need label editing |
| **Invite system** | Token-based, 10min expiry, join requests with IP/email capture | Approval UI only in Inbox; needs dedicated approval screen with Name/Email/IP |
| **User roles** | `instance_admin` + `company_membership` with `membershipRole: "member"` | No Admin/Member distinction at company level; no permission enforcement per role |
| **User profile** | No profile page exists; sign-out only in instance settings | Need full profile page with name, email, avatar upload, role display |
| **Avatar system** | `authUsers.image` column exists but no upload mechanism | Need file upload for profile pictures |

## Workstream 1: Custom Status Workflows

**Goal:** Let company admins define custom issue statuses (create, rename, reorder, color, icon) per company — similar to Linear's workflow customization.

### 1.1 Database Schema

**New table: `company_statuses`**

```
packages/db/src/schema/company_statuses.ts

company_statuses:
  id            uuid PK default random
  company_id    uuid FK companies.id NOT NULL
  name          text NOT NULL                    -- e.g. "In QA", "Deployed"
  slug          text NOT NULL                    -- kebab-case, used as status value
  category      text NOT NULL                    -- "backlog" | "unstarted" | "started" | "completed" | "cancelled"
  color         text NOT NULL                    -- hex color, e.g. "#f59e0b"
  icon          text                             -- optional icon identifier
  position      integer NOT NULL default 0       -- ordering within category
  is_default    boolean NOT NULL default false   -- one per category can be default
  created_at    timestamptz default now
  updated_at    timestamptz default now

Indexes:
  UNIQUE (company_id, slug)
  INDEX  (company_id, category, position)
```

**Key design decisions:**
- The `category` field maps custom statuses to one of 5 semantic buckets (backlog, unstarted, started, completed, cancelled). This preserves the state machine logic (transition guards, side effects like `started_at`, `completed_at`, `cancelled_at`) while allowing unlimited custom statuses within each bucket.
- The `issues.status` column changes from storing hardcoded enum strings to storing `company_statuses.slug` values.
- On company creation, seed the 7 default statuses mapped to categories:
  - `backlog` → category `backlog`
  - `todo` → category `unstarted`
  - `in_progress` → category `started`
  - `in_review` → category `started`
  - `blocked` → category `unstarted`
  - `done` → category `completed`
  - `cancelled` → category `cancelled`

### 1.2 Migration Strategy

1. Create `company_statuses` table
2. For each existing company, insert the 7 default status rows
3. Existing `issues.status` text values already match the default slugs — no data migration needed
4. Add a `status_category` computed/join column to issue queries for state machine logic

### 1.3 Shared Package Changes

**File: `packages/shared/src/constants.ts`**
- Add `STATUS_CATEGORIES = ["backlog", "unstarted", "started", "completed", "cancelled"] as const`
- Keep `ISSUE_STATUSES` as fallback defaults, but mark as `DEFAULT_ISSUE_STATUSES`
- Add `StatusCategory` type

**File: `packages/shared/src/types/issue.ts`**
- Add `CompanyStatus` interface: `{ id, companyId, name, slug, category, color, icon, position, isDefault }`

**File: `packages/shared/src/validators/issue.ts`**
- Add `createCompanyStatusSchema`: name (1-32 chars), category (enum), color (hex), icon (optional)
- Add `updateCompanyStatusSchema`: name, color, icon, position (all optional)
- Update issue validators to accept any string for status (validated server-side against company statuses)

### 1.4 Server Changes

**New routes:**
- `GET    /api/companies/:companyId/statuses` — list all statuses for company (ordered by category + position)
- `POST   /api/companies/:companyId/statuses` — create custom status (Admin only)
- `PATCH  /api/statuses/:statusId` — update name/color/icon/position (Admin only)
- `DELETE /api/statuses/:statusId` — delete status (Admin only, must not have issues assigned, or require target migration status)
- `POST   /api/companies/:companyId/statuses/reorder` — batch reorder positions

**Service changes (`server/src/services/issues.ts`):**
- Transition guard: validate that requested status slug exists for the company
- Map `category` for side-effect logic:
  - `started` category → set `started_at`
  - `completed` category → set `completed_at`
  - `cancelled` category → set `cancelled_at`
- Transition rules based on category flow: `backlog → unstarted → started → completed/cancelled`
- Kanban board columns map to company statuses

### 1.5 UI Changes

**New: Company Status Settings section in `CompanySettings.tsx`**
- Drag-to-reorder status list grouped by category
- Inline edit for name, color picker, icon selector
- Add/delete status buttons per category
- Warning when deleting a status that has active issues

**Updated: `StatusIcon.tsx`, `StatusBadge.tsx`**
- Fetch company statuses from API instead of using hardcoded list
- Use custom colors from `company_statuses.color`
- New hook: `useCompanyStatuses(companyId)` with React Query caching

**Updated: `KanbanBoard.tsx`**
- Columns driven by `company_statuses` query instead of hardcoded `boardStatuses`
- Respect position ordering

**Updated: `IssuesList.tsx`**
- Status filter options from company statuses
- Status grouping/sorting uses custom status order

**Updated: `IssueProperties.tsx`**
- Status dropdown populated from company statuses

## Workstream 3: User Profile & Settings Page

**Goal:** Create a dedicated profile page where users can view and edit their name, email, profile image, and see their role.

### 3.1 Profile Image Upload

**Server changes:**

- `POST /api/auth/profile/avatar` — multipart file upload endpoint
  - Validates file: max 2MB, image types only (jpeg, png, gif, webp)
  - Stores via the existing asset storage system (`local_disk` or `s3`)
  - Updates `authUsers.image` with the asset URL/path
  - Returns the new image URL

- `DELETE /api/auth/profile/avatar` — remove avatar, set `authUsers.image` to null

- `PATCH /api/auth/profile` — update profile fields
  - Accepts: `name` (string, 1-100 chars)
  - Does NOT allow email change in V1 (security consideration — email change requires verification flow)

- `GET /api/auth/profile` — return current user profile
  - Returns: `{ id, name, email, image, role, companies: [{ id, name, membershipRole }] }`
  - `role` derived from `instance_user_roles` (admin) or default (member)

## Workstream 5: Admin/Member Role System

**Goal:** Implement a two-tier company-level role model (Admin + Member) with proper permission enforcement.

### 5.1 Role Definitions

| Capability | Admin | Member |
|-----------|-------|--------|
| View issues, comments, agents, projects, goals | ✅ | ✅ |
| Create/edit issues | ✅ | ✅ |
| Post comments | ✅ | ✅ |
| Assign issues (to self or others) | ✅ | ✅ (self + available agents) |
| Create projects & goals | ✅ | ✅ |
| **Manage company settings** | ✅ | ❌ |
| **Create/manage custom statuses** | ✅ | ❌ |
| **Create/edit/delete labels** | ✅ | ❌ |
| **Invite new users** | ✅ | ❌ |
| **Approve/reject join requests** | ✅ | ❌ |
| **Create/manage agents** | ✅ | ❌ |
| **Pause/resume/terminate agents** | ✅ | ❌ |
| **Set budgets** | ✅ | ❌ |
| **Manage member roles** | ✅ | ❌ |
| **Archive/delete company** | ✅ (instance_admin) | ❌ |

### 5.2 Implementation Approach

The `company_memberships.membershipRole` column already exists. Currently it's always `"member"`. The plan:

1. **Update shared constants:**
   ```ts
   export const COMPANY_ROLES = ["admin", "member"] as const;
   export type CompanyRole = (typeof COMPANY_ROLES)[number];
   ```

2. **Server middleware:**
   - Update `assertCompanyPermission()` to check role-based permissions
   - Create `requireCompanyAdmin(req, companyId)` helper
   - Admin-only routes get the admin check; member routes pass through with basic membership check

3. **Company creation:**
   - The creating user is automatically set as `admin` for that company (update `ensureMembership` call)

4. **Instance admin override:**
   - `instance_admin` always has admin access to all companies (preserve existing behavior)

5. **Role management UI:**
   - In Company Settings → Members section:
     - List all members with their role (Admin/Member)
     - Admin can promote Member → Admin or demote Admin → Member
     - Cannot demote yourself if you're the last admin
   - Visual role badge next to user names throughout the UI

## Key Design Principles

1. **Backwards compatible** — Existing data (issues, comments, labels) continues to work without migration issues. Default statuses are seeded on company creation.
2. **Progressive disclosure** — Members see a clean, focused interface. Admin settings are behind a settings page, not cluttering the main workflow.
3. **Security-first** — Invite tokens are short-lived, hashed at rest. Role checks happen server-side (never trust the client). Rate limiting on sensitive endpoints.
4. **Consistent with existing patterns** — Use the same Drizzle schema conventions, API route patterns, React Query hooks, and shadcn/ui components already in the codebase.
5. **Minimal new dependencies** — No new npm packages needed. All features built with existing tools (Tailwind, Radix, React Query, Drizzle).
